### 1.1 Android 16 BLE 修复（v1.0 — 已发布）

飞智散热器开发者工具在 Android 16 上完全无法连接散热器，根源是 4 层嵌套 Bug：

| 层级 | 问题 | 修复方式 | 文件位置 |
|------|------|---------|---------|
| #1 控制器层 | 设备连接后 `stopScan()` 未调用，导致扫描一直运行 | 钩 `AbstractBluetoothController.onDeviceConnected` → 调 `stopScan()` | `MainHook.java` |
| #2 ViewModel 层 | `BluetoothViewModel.onDeviceConnected` 的 LiveData 未及时更新 | 钩该方法 → `postValue()` 更新 `_bluetoothNameLiveData`、`_macLiveData`、`_connectLiveData` | `MainHook.java` |
| #3 UI 层 | 智能温控 UI 闪烁（循环触发） | `convertFromDevice()` 不该用（创建 null 字段的 WaspWingInfo → 触发 `onDeviceInfoUpdate` → `setExperimentalRunMode` → 状态循环） | `MainHook.java` |
| #4 Android 16 | `checkBluetoothPermission()` 在 BLE 回调线程返回 false → `discoverServices()` 永不被调用 | 钩该方法 → 强制返回 `true` | `MainHook.java` |

**关键发现**：`discoverServices()` 没被调用的根因是 Android 16 (API 36) 的权限检查行为变化。

### 1.4 FIFO 唤醒机制

**废弃轮询方案**，采用 **FIFO（命名管道）事件驱动**：

```
模块 → FIFO → C 守护进程：
  '1' = BLE 已连接/重连，请求开始控制
  '0' = BLE 断联，请求停止控制
```

模块（MainHook.java）写入时机：
| 事件 | 写入 | 原因 |
|------|------|------|
| `AbstractBluetoothController.onDeviceConnected`（BLE 连接成功） | `'1'` | **主要唤醒信号** |
| `BluetoothGatt.disconnect()`（BLE 断联） | `'0'` | **主要停止信号** |
| `B6ExperimentalActivity.onResume()` + BLE 已连接 | `'1'` | daemon 晚启动时的兜底 |
| `B6ExperimentalActivity.onResume()` + BLE 未连 | 不写 | 等待连接回调 |
| `B6ExperimentalActivity.onPause()` | 不写 | App 切后台不影响散热控制 |

C 守护程序状态机：
```
启动 → peek FIFO
  ├─ 有 '1' → 直接进工作模式
  └─ 无数据 → blocking read(FIFO)（CPU 0%）

收到 '0' → 记录时间，继续阻塞等
收到 '1' → 检查断联时间
  ├─ ≥60 秒 → reset_state()（重读电池温度算初始档位，强制下发）
  └─ <60 秒 → 正常恢复

工作模式：每5秒 main_loop() → peek FIFO → 有 '0' 回阻塞
```

FIFO 路径：`/data/local/tmp/b6x_fifo`（权限 0666）

### 4.2 FIFO 阻塞等待（C — tempctrl.c）

```c
static int wait_for_fifo(void) {
    fifo_fd = open(FIFO_PATH, O_RDONLY);  // 阻塞模式
    char c;
    while (running) {
        int n = read(fifo_fd, &c, 1);  // ★ 内核挂起，CPU 0%
        if (n == 1 && c == '1') {
            if (disconnect_time > 0 && 
                (time(NULL) - disconnect_time) >= 60)
                reset_state();
            disconnect_time = 0;
            return 1;
        } else if (n == 1 && c == '0') {
            disconnect_time = time(NULL);
        } else if (n == 0) {
            close(fifo_fd);
            fifo_fd = open(FIFO_PATH, O_RDONLY);
        }
    }
    return 0;
}
```

## 五、需要避的坑

### 🔴 严重（会导致失败）

1. **C 代码顺序依赖** — `reset_state()` 被 `wait_for_fifo()` 调用，但定义在后面。NDK clang 的 C99 模式严格检查此问题。**每次加新函数时检查前向声明。**
   ```c
   // 需要在 FIFO 函数前加：
   static void reset_state(void);  // 前向声明
   ```

2. **`#include <stdarg.h>` 不能少** — `write_log()` 用了 `va_list`，该头文件在 NDK clang 下不是隐式包含的。

3. **不要用 `convertFromDevice()`** — 它会创建全默认值的 `WaspWingInfo`（`experimentalRunModeValue=null`），触发 `App.onDeviceInfoUpdate` → `setExperimentalRunMode(true, coldLevel+1)` 状态循环，导致 UI 闪烁。

### 🟡 需要注意

4. **FIFO 权限** — `mkfifo(FIFO_PATH, 0666)` 确保 app 进程（非 root）也能写。

5. **FIFO 断连处理** — 当所有写端关闭（如模块被杀），`read()` 返回 0 而不是阻塞。必须处理此情况：`close()` 后重新 `open()`。

6. **CI 中不要用 `setup-ndk` action 的 `$ANDROID_NDK_HOME`** — 该环境变量在某些版本设置异常。改用：
   ```yaml
   - name: 下载 NDK
     run: |
       wget -q https://dl.google.com/android/repository/android-ndk-r27c-linux.zip
       unzip -q android-ndk-r27c-linux.zip
       mv android-ndk-r27c /opt/ndk
   # 然后在编译步骤中用绝对路径：
   CC=/opt/ndk/toolchains/llvm/prebuilt/linux-x86_64/bin/aarch64-linux-android21-clang
   ```

7. **Git 分支管理** — 不要同时用 `main` 和 `master`，默认分支是 `main`。推荐：
   ```bash
   git push origin master:main --force   # 将 master 覆盖到 main
   ```
   然后删掉 master 分支，统一用 main。

### 🟢 小问题

8. **编译产物 TLS 对齐** — `patch_tls.py` 修复 PT_TLS 段对齐。如果编译时没有 TLS 段，脚本返回非零退出码，需要 `|| echo "skipped"` 容忍。

9. **不要推送 release tag 到 CI 直到测试通过** — 测试阶段用 `workflow_dispatch` 触发编译。

10. **`2>&1` 重定向位置** — 在 Bash 中 `command 2>&1` 是正确的，但如果写成 `command 2>&1` 后跟 `|| true`，要注意 stdout/stderr 合并的副作用。
