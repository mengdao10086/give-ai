## 1. 原始问题

### 1.1 现象

打开飞智散热器 App，散热器已配对并自动回连蓝牙，但 UI 持续显示"扫描中"，无法进入控制界面。设备实际上已通过 ACL 层连接（蓝牙设置中可见已连接），但 App 的扫描动画永不停止。

### 1.2 App 架构

```
com.example.extool                        ← 应用层
  ├── App.kt                              应用入口
  ├── B6ExperimentalActivity.kt           B6X 实验模式界面 (DataBinding)
  ├── BluetoothViewModel.kt               蓝牙抽象 ViewModel
  └── WaspWingViewModel.kt                散热器具体 ViewModel

com.flydigi.sdk.bluetooth                  ← 蓝牙 SDK
  ├── AbstractBluetoothController.kt      核心控制器（扫描/连接/广播接收）
  ├── LeDataInteractionController.kt      BLE GATT 交互控制器
  └── AbstractDataInteractionController.kt 数据交互基类

com.flydigi.sdk.waspwing                   ← 散热器协议层
  ├── WaspWingManager.kt                  管理器单例
  ├── WaspWingDataInteractionController.kt BLE 协议实现
  └── WaspWingInfo.kt                     设备信息数据类 (Parcelable)
```

### 1.3 关键数据流

```
WaspWingInfo (设备状态数据类, ~31个构造参数)
  ├── connected          → DataBinding 切换扫描/连接 UI
  ├── runMode            → 运行模式 (0=固定功率, 1=智能温控, ...)
  ├── realWindLevel      → 实时风速档位
  ├── realColdLevel      → 实时制冷档位
  ├── temperature        → 当前温度
  ├── experimentalRunModeValue → 实验模式值 (智能温控核心字段)
  ├── deviceCode         → 设备型号代码 ("b6", "b6x", "b5", ...)
  └── ...                → 其他字段驱动 UI 控件状态

LiveData 观察链:
  WaspWingViewModel._waspWingInfo (MutableLiveData<WaspWingInfo>)
    → B6ExperimentalActivity 的 DataBinding (ActivityExperimentalB6Binding)
      → executeBindings() 检查各字段 → 控制 UI 渲染
```

正常连接流程：
```
startScan() → 发现设备 → onDeviceFound() → stopScan() → connect(device)
  → GATT 连接 → onGattConnected → discoverServices()
  → onServicesDiscovered → initCharacteristic → 注册通知
  → onCharacteristicChanged (0x13 数据包) → onReceiveDataFromDevice
  → deviceInfo 字段更新 → onDeviceInfoUpdate(waspWingInfo)
  → ViewModel._waspWingInfo.postValue(info) → DataBinding 更新 UI
```

## 2. 第一阶段：APK 直接修改尝试（失败）

### 2.1 尝试的修改方案

| 方案 | 目标 | 结果 |
|------|------|------|
| Fix #1 - ACL_CONNECTED 加 stopScan | BroadcastReceiver smali 修改 | 失败 - smali 往返后 DEX 崩溃 |
| Fix #2 - 废弃 startLeScan 替换 | AbstractBluetoothController smali | 失败 - 同上 |
| Fix #3 - registerReceiver 加导出标志 | 构造函数 smali | 失败 - 闪退 |
| Fix #4 - onDeviceConnected 加 stopScan | BluetoothViewModel smali | 失败 - 未实际部署 |

### 2.2 失败根因

smali 2.5.2 编译后 DEX 在 SDK 36 上运行不稳定，即使零修改的 smali 往返也会在蓝牙扫描时崩溃。这是 smali 工具链对 DEX 040 格式支持不完善导致的。

### 2.3 结论

放弃 APK 修改路线，转向 LSPosed 运行时 Hook 方案。

## 3. 第二阶段：LSPosed 模块初步修复

### 3.1 初始修复（修复 #1 + #2）

**修复 #1 — 控制器层保底**：钩 `AbstractBluetoothController.onDeviceConnected(BluetoothDevice)`  
→ after 阶段调 `stopScan()` + 设置 `inScanning=false`

**修复 #2 — ViewModel 层数据更新**：钩 `BluetoothViewModel.onDeviceConnected(BluetoothDevice)`  
→ before 阶段：
- 调 `stopScan()`
- 更新 `_bluetoothNameLiveData.postValue(name)`
- 更新 `_macLiveData.postValue(addr)`
- 更新 `_connectLiveData.postValue(true)`
- **更新 `_waspWingInfo`**：用 `WaspWingInfo.convertFromDevice(device)` 创建新实例并 postValue

### 3.2 踩坑 #1：NoSuchFieldError

```
java.lang.NoSuchFieldError: No field _connectedLiveData in ...WaspWingViewModel
```

- **原因**：App 的 `WaspWingViewModel` 没有 `_connectedLiveData`，只有 SDK 的 `WaspwingViewModel` 有。
- **修复**：在 catch 块中捕获 `Throwable` 而非 `Exception`，使 Error 不传播。

### 3.3 踩坑 #2：_waspWingInfo 始终为 null

```
waspWingInfo 为 null，从 device 创建
```

- **诊断**：ACL_CONNECTED 时 `_waspWingInfo` 尚未被初始化（正常流程需要 GATT 数据到达后才设置）
- **修复**：使用 `convertFromDevice(device)` 工厂方法创建已连接的 WaspWingInfo

### 3.4 踩坑 #3：双 GATT 连接

```
[诊断] connect() 进入 (第一次)
[诊断] connect() 进入 (第二次) — 重复！
```

- **原因**：修复 #2 创建 WaspWingInfo 后额外调用了 `connectGattWith()`，但 SDK 已在 `onDeviceFound` → `connect()` 中自动发起了 GATT 连接
- **修复**：去掉冗余的 `connectGattWith()` 调用

## 4. 第三阶段：GATT 服务发现之谜

### 4.1 症状

UI 显示"已连接"且 MAC 地址正确，但所有控制按钮灰化不可操作。检查发现：
- `onGattConnected` 触发 ✓
- `services=0`（`gatt.getServices()` 返回空列表）
- `onServicesDiscovered` 永不触发 ❌

### 4.2 诊断日志

```
[诊断] WaspWingDataInteractionController.onGattConnected 已触发
[诊断]   services=0
// ← 注意：没有 onServicesDiscovered 的日志！
```

### 4.3 深挖调用链

检查 `LeDataInteractionController$mGattCallback$1.onConnectionStateChange`：

```smali
.method public onConnectionStateChange(BluetoothGatt, status, newState)V
    if newState == 2 (CONNECTED):
        this$0.onGattConnected(gatt)    ← 调 onGattConnected
    // 注意：这里不调 gatt.discoverServices()！
.end method
```

`discoverServices()` 在 `LeDataInteractionController.onGattConnected()` 中被调用：

```smali
.method protected onGattConnected(BluetoothGatt gatt)V
    checkBluetoothPermission()          ← 返回 false 则提前 return！
    if !granted: return-void
    ...
    gatt.discoverServices()             ← 只有权限通过后才会执行
.end method
```

**关键发现**：使用 Xposed 钩 `BluetoothGatt.discoverServices()` 本身——

```
// 日志中完全没有 discoverServices() 被调用的记录！
```

证明 `discoverServices()` 从未被执行。问题锁定在 `checkBluetoothPermission()` 返回 false。

### 4.4 根因：Android 16 权限检查

`checkBluetoothPermission()` 实现逻辑：

```smali
private checkBluetoothPermission()Z:
    if (SDK_INT > 30):
        // Android 12+ 检查 BLUETOOTH_CONNECT
        return PermissionChecker.checkSelfPermission(BLUETOOTH_CONNECT) == GRANTED
    if (SDK_INT <= 30):
        // Android 11- 检查 BLUETOOTH
        return Context.checkSelfPermission(BLUETOOTH) == GRANTED
    return false
```

在 Android 16 (SDK 36) 上：
- SDK_INT > 30 → 检查 `BLUETOOTH_CONNECT`
- 该权限在 `connect()` 中通过（主线程，有上下文权限）
- 但在 `onGattConnected()` 中失败（BLE 回调线程，PermissionChecker 返回不一致）

> **为什么 `connect()` 能通过而 `onGattConnected()` 不能？**
> 目前推测与调用线程有关。`PermissionChecker.checkSelfPermission()` 在 Android 16 上可能对线程上下文敏感。
> 实际原因为 Android 14+ 对 `BLUETOOTH_CONNECT` 权限的检查更加严格，只有 manifest 中声明了 `<uses-permission>` 的 app 才能通过。
> 原 APK 编译时 `targetSdkVersion` 较低，未声明此权限。
> `connect()` 中通过可能是因为系统对 `connectGatt()` 的权限有宽松处理。

### 4.5 修复 #4：强制权限通过

```java
XposedHelpers.findAndHookMethod(leCtrl, "checkBluetoothPermission",
    new XC_MethodHook() {
        @Override
        protected void afterHookedMethod(MethodHookParam param) {
            param.setResult(true);    // 强制返回 true
        }
    });
```

此修正后，完整 GATT 链路恢复：
```
checkBluetoothPermission → true ✓
discoverServices() → true ✓
onServicesDiscovered → status=0 ✓
onCharacteristicChanged → 已触发 ✓
App.onDeviceInfoUpdate → connected=true, _waspWingInfo=有值 ✓
```

### 4.6 安全考量

强制蓝牙权限通过是否安全？
- 模块运行在用户已 root 的设备上（KernelSU）
- `checkBluetoothPermission()` 仅用于 SDK 内部的函数级保护，不影响系统权限模型
- 该钩子只作用于目标 App 进程内，不影响其他应用
- 本质上等效于：原 App 若声明了 `BLUETOOTH_CONNECT` 应有的行为

## 5. 第四阶段：智能温控 UI 闪烁

### 5.1 症状

设备连接正常，控制操作可用。但：
- 初次进入 B6ExperimentalActivity 时，若散热器处于**智能温控模式**，模式指示器一直闪烁
- 切换为固定功率模式再切回→恢复正常
- 初始即为固定功率模式→不会闪烁
- 日志异常长（每秒大量输出）

### 5.2 日志分析

```
02:25:20.863 - onServicesDiscovered
02:25:20.898 - onCharacteristicChanged #1
02:25:21.806 - App.onDeviceInfoUpdate #1 (connected=true)
02:25:21.806 - onCharacteristicChanged #2
02:25:21.918 - App.onDeviceInfoUpdate #2
02:25:22.809 - App.onDeviceInfoUpdate #3
02:25:22.809 - onCharacteristicChanged #3
... 持续每秒刷新 ...
```

智能温控模式下，设备每秒发送一次温度数据包 (0x13)，每次触发 DataBinding 重渲染。

### 5.3 根因追踪

查看 `App.onDeviceInfoUpdate`（WaspWingViewModel 的实现）：

```kotlin
// 伪代码
fun onDeviceInfoUpdate(info: WaspWingInfo) {
    _waspWingInfo.postValue(info)   // 触发 DataBinding
    
    // 智能温控自修复逻辑：
    if (info.experimentalRunModeValue == null 
        || info.experimentalRunModeValue != info.realColdLevel + 1) {
        // 重新发送智能温控模式指令
        WaspWingManager.setExperimentalRunMode(true, realColdLevel + 1, ...)
    }
}
```

问题链路：
1. **修复 #2** 用 `convertFromDevice(device)` 创建 WaspWingInfo
2. `convertFromDevice` 创建的实例所有字段为默认值 → `experimentalRunModeValue = null`
3. 该对象被 `postValue` 到 `_waspWingInfo`
4. 正常数据流到来后，`onDeviceInfoUpdate` 检查到 `experimentalRunModeValue == null`
5. → 调用 `setExperimentalRunMode(true, coldLevel+1)` 发送指令
6. 设备响应指令 → 触发新的 `onCharacteristicChanged` → 又触发 `onDeviceInfoUpdate`
7. 若 `coldLevel` 在温度数据包中更新 → `experimentalRunModeValue != coldLevel+1` → 再次触发
8. 每次 `_waspWingInfo.postValue()` → DataBinding 执行 `executeBindings()` → 模式指示器重渲染 → 视觉上闪烁

### 5.4 修复方案

去掉修复 #2 中的 `convertFromDevice` 调用：

```java
// 之前：_waspWingInfo 为 null 时创建新对象
if (currentInfo != null) {
    // 更新 connected
} else {
    // ❌ 删掉这整个分支
    WaspWingInfo newInfo = convertFromDevice(device);
    _waspWingInfo.postValue(newInfo);    // 导致闪烁
}

// 之后：为 null 就不处理，等正常数据流填充
if (currentInfo != null) {
    XposedHelpers.callMethod(currentInfo, "setConnected", true);
    XposedHelpers.callMethod(waspInfoMd, "postValue", currentInfo);
}
// currentInfo == null：跳过，正常 GATT 数据流约 1s 后到达
```

因为修复 #4 已保证 GATT 服务发现和数据流正常运转，`_waspWingInfo` 会在约 1 秒内被正常数据填充。无需提前创建。

### 5.5 精简日志

同时移除了高频诊断钩（每秒触发多次）：
- `onCharacteristicChanged` — 每温控数据更新触发一次
- `App.onDeviceInfoUpdate` — 每数据更新触发一次
- `checkBluetoothPermission` before log — 每次写入操作都触发

保留低频诊断钩（仅在连接阶段触发一次）：
- `connect()` — 连接时
- `discoverServices()` — 服务发现时
- `onServicesDiscovered` — 服务发现完成时
- `onGattConnected` — GATT 连接时
- `SDK.onDeviceInfoUpdate` — 首次数据更新时

## 7. 技术总结

### 7.1 四层 Bug 链

```
第1层：扫描不停 → 简单，stopScan() 解决
  ↓
第2层：UI 不更新 → LiveData 手动填充解决
  ↓  
第3层：GATT 不完整 → Android 16 权限暗坑，checkBluetoothPermission 返回 false
  ↓
第4层：智能温控闪烁 → convertFromDevice 创建全默认值对象触发状态循环
```

### 7.2 关键技术发现

| 发现 | 详情 |
|------|------|
| `ACL_CONNECTED` 已有 stopScan | 反编译确认 BroadcastReceiver 第 349 行已调 `stopScan()` |
| `checkBluetoothPermission` 在 Android 16 异常 | SDK 内部权限检查在 BLE 回调线程返回 false |
| `convertFromDevice` 有副作用 | 创建的 WaspWingInfo 所有字段为默认值（0/null/false） |
| `App.onDeviceInfoUpdate` 中有自修复逻辑 | 若 `experimentalRunModeValue` 异常则重新发送智能温控指令 |
| 智能温控模式每秒发一次数据 | 设备 0x13 数据包约 1 秒间隔，DataBinding 每次重渲染 |

### 7.3 给 Android 16 BLE 开发者的注意事项

1. **权限模型**：Android 12+ 需要 `BLUETOOTH_CONNECT` 运行时权限才能调用 `discoverServices()`。声明 `<uses-permission>` 并在运行时请求。Android 14+ 对权限检查更严格。
2. **废弃 API**：`BluetoothAdapter.startLeScan()` 已废弃多年，在 SDK 36 上行为不确定。
3. **GATT 回调线程**：`onConnectionStateChange` 在 BLE 回调线程上执行，此线程上的权限状态可能与主线程不一致（Android 16 特有行为？）。
4. **`registerReceiver`**：Android 14+ 要求指定 `RECEIVER_EXPORTED` 或 `RECEIVER_NOT_EXPORTED`，否则注册可能失败。
5. **DataBinding 与 LiveData**：`postValue()` 即使传入同一对象引用也会触发观察者，注意高频率更新对 UI 的影响。
