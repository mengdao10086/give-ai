# tempctrl 智能温控守护程序 — 设计说明

对应实现：[tempctrl.c](tempctrl.c)
配置文件：[magisk模块框架/profile.conf](magisk模块框架/profile.conf)
本文件只说明运行逻辑，不追求代码级精细。

---

## 整体架构

```
 ┌──────────────────┐               ┌──────────────────┐
 │ LSPosed 模块     │   am broadcast │ tempctrl 守护进程 │
 │ (MainHook.java)  │ ←────────────│ (本 C 程序)      │
 │                  │   mode/temp/  │                   │
 │ 收到广播 → 调用   │   windOC/    │ pgrep → 检测 App  │
 │ WaspWingManager  │   coldOC/    │ 存活状态         │
 │ .setRunMode()    │   windLevel   │ 每 5s main_loop() │
 │      ↓           │               │                   │
 │ BLE → 散热器     │               │ 热重载 profile.   │
 └──────────────────┘               │ conf 配置         │
                                    └──────────────────┘
```

**通信方式变更（v2.0）**：~~FIFO 写入~~ → 已废弃，改为 `pgrep` 检测 App 进程存活。
tempctrl 不再通过 FIFO 接收唤醒/休眠信号，而是每 5 秒检查进程是否存在。

## 配置文件系统 (profile.conf)

tempctrl 支持运行时配置文件，所有关键阈值和参数均可外部修改，无需重新编译。

### 配置路径自动检测

启动时通过 `/proc/self/exe` 获取自身路径，在同目录下查找 `profile.conf`。

找不到 → 全部使用代码内默认值，仅记一条日志。

也支持 `--config` 参数手动指定：
```shell
tempctrl --config /自定义路径/profile.conf
```

### 配置格式

```
# 注释以 # 开头
KEY=VALUE

# 例：
BATT_BASELINE=350
BATT_ZONE_1=7
```

不认识的 key 静默跳过；每个值有范围保护，非法值不会导致进程崩溃。

### 热重载

每轮 `main_loop()`（5 秒间隔）开头 `stat()` 检查配置文件的修改时间（mtime），变了就自动重读所有参数并记日志。**修改配置无需重启 daemon**。

### 可配置参数一览

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `CONFIG_ENABLED` | 1 | 配置文件总开关（0=全部默认） |
| `BATT_BASELINE` | 350 | 电池基准温度 (0.1°C) |
| `BATT_ZONE_1` | 7 | 第一区间阈值 (0.1°C)，死区 |
| `BATT_ZONE_2` | 20 | 第二区间阈值 (0.1°C) |
| `OVERRIDE_MAX` | 8 | 趋势豁免最大连续次数 |
| `PEAK_DAMP_THRESHOLD` | 3 | 2°C 外严格豁免阈值 (0.1°C) |
| `PEAK_DAMP_INNER_THRESHOLD` | 5 | 2°C 内峰值反补触发阈值 (0.1°C) |
| `PEAK_DAMP_INNER_ADJUST` | 2 | 2°C 内峰值反补档位数 |
| `PEAK_DAMP_OUTER_THRESHOLD` | 3 | 2°C 外峰值反补触发阈值 (0.1°C) |
| `PEAK_DAMP_OUTER_ADJUST` | 1 | 2°C 外峰值反补档位数 |

## 温度读取

- **电池温度**：`/sys/class/power_supply/battery/temp`
  - 原生 0.1°C 精度，如 350 = 35.0°C

- **CPU 温度**：`/sys/class/thermal/thermal_zone[N]/temp`
  - 原始值 m°C，÷100 转 0.1°C，如 75321 → 753 (=75.3°C)
  - 首次调用时自动扫描 zone0~99，记录所有能读到有效值的 zone
  - 后续只扫描已记录的 zone，取最高值
  - 一阶低通滤波平滑（系数由 `CPU_FILTER_ALPHA` 配置，默认 α=0.25）

## 主循环逻辑

`main_loop()` 每 **5 秒**执行一次：

0. **检查配置更新** → `stat()` 检测 `profile.conf` 修改时间，变了就热重载
1. **紧急干预** → 读 CPU 温度 → 低通滤波 → 判定紧急等级 (0~3)，设 `forced_min_level`
2. **电池控制** → 读电池温度 → 计算温度变化量 + 常规档位调整量
   - 温度读数不变则跳过
   - **合并逻辑**（趋势豁免 + 峰值反补合为一体）：
     - 变动 ≤0.3°C 且趋势反向 → **趋势豁免**（跳过本次常规调整），首次触发记日志
     - 变动 >0.3°C → **峰值反补**（直接改档位，跳过常规调整）：
       - 2°C 内 0.3~0.5°C → 反补 1 档
       - 2°C 内 >0.5°C → 反补 PEAK_DAMP_INNER_ADJUST 档（默认 2）
       - 2°C 外 >0.3°C → 反补 PEAK_DAMP_OUTER_ADJUST 档（默认 1）
   - 未被豁免/反补时执行常规升降档（偏差三区策略）
3. **合最终档位** = `max(battery_fan_level, forced_min_level)`
4. **下发**控制参数（去重）
5. **同步** `battery_fan_level = final_level`（电池以下次以上次实际档位为基础）
6. **退出紧急限制**：如果刚退出紧急等级且档位过高 → 硬 cap 到 `EMERG_FORCED_X`
   - 大电流充电（`current_now < -2000mA`）时跳过此限制
7. **pgrep 检测** App 进程存活 → 失联则等待复活后强制重发

## 进程检测机制

使用 **双重检测**：pgrep 进程存在检查 + 状态文件 mtime 心跳检查。只要有一个出问题就算断联。

```
 启动 → 创建 status 文件（0666，所有用户可写）
      → 等待 App 进程出现（pgrep 通过即可进入）
 
 工作模式中，每 5 秒：
   main_loop() → 决策 → 下发
   
   1. 读 status 文件内容 → BLE=0/1（模块上报的 BLE 连接状态）
   
   2. 双重存活检测：
      ├─ pgrep -f com.flydigi.waspwing.experimental  → 进程存在?
      └─ stat(status 文件).mtime 距现在 ≤ 16秒？     → 模块在写心跳?
      
      ├─ 两者都 OK → 存活
      └─ 任一失败  → App 死了 → 记录失联时间
                          ↓
                      等待复活子循环（每 5 秒检测）
                          ↓
                      App 重新出现 → 强制重发当前档位
                          ↓
                      检查断联时间
                        ├─ ≥ DISCONNECT_RESET_SEC → reset_state()
                        └─ < DISCONNECT_RESET_SEC → 正常恢复

 模块侧（LSPosed）：
   ────────────────────────────────
   onDeviceConnected / disconnect 钩子 → 更新 bleConnected
   每 5 秒线程：FileOutputStream 写 "BLE=0/1\n" 到 status 文件
   （直接 Java 文件 I/O，不走 su，无 KSU 命名空间问题）
```

### 文件详情

| 项目 | 说明 |
|------|------|
| 路径 | `/data/local/tmp/tempctrl.status` |
| 创建者 | tempctrl 启动时 `fopen("a")` + `chmod 0666` |
| 写入者 | LSPosed 模块每 5 秒覆写 `BLE=0\n` 或 `BLE=1\n` |
| 读取者 | tempctrl `main_loop` 后 `read_status_ble()` |
| 心跳超时 | 16 秒（STATUS_TIMEOUT，写死后约 3 轮检测判死） |

### 双检测互补

| 场景 | pgrep | 文件 mtime | 结果 |
|------|-------|-----------|------|
| App 正常运行，BLE 已连 | ✅ | ✅（新鲜 BLE=1） | 存活 |
| App 正常运行，BLE 断联 | ✅ | ✅（新鲜 BLE=0） | 存活（但不控温） |
| App 被 kill | ❌ | ❌（不再更新） | 断联 |
| App 卡死/ANR | ✅（进程在） | ❌（无写入） | 断联 |
| 文件被意外删除 | ✅ | ❌（stat 失败） | 断联（daemon 会重建） |

## 电池控制细节

- **基准温度**：35.0°C（`BATT_BASELINE=350`，可配置）
- **三级策略**（diff = 当前温度 - 基准温度）：
  - `|diff| ≤ BATT_ZONE_1` → 不动，属于正常范围
  - `BATT_ZONE_1 < |diff| ≤ BATT_ZONE_2` → 升/降 1 档
  - `|diff| > BATT_ZONE_2` → 升/降 2 档

- **档位继承同步**：
  - 每次 `main_loop` 末尾 `battery_fan_level = final_level`
  - 电池控制始终从上次实际下发的档位开始调整
  - 紧急退出后不会暴跌——电池继承了紧急期间的档位
  - 退出紧急限制在同步之后执行：如果档位过高则硬 cap

- **温度趋势豁免（融合峰值时间检测）**：
  - **基准 2°C 以内**：趋势反向即豁免（降温中要升档 → 跳过，升温中要降档 → 跳过）
  - **超过 2°C**：仅当单次变动 >PEAK_DAMP_THRESHOLD（默认 0.4°C）才豁免，温度偏离较大时更积极响应
  - 最多连续豁免 `OVERRIDE_MAX` 次（默认 8 次 ≈ 40 秒），之后强制执行
  - 最高/最低档位时不记日志（避免刷屏）
  - 防止温度小幅波动导致反复升降档

- **峰值过冲抑制**：
  - **基准 2°C 内**：单次变动 >0.5°C（5 单位）→ 反向补偿 **2** 档
    - 急升 → +2 档（提前压制升温）
    - 急降 → -2 档（防止过冷）
  - **基准 2°C 外**：单次变动 >0.3°C（3 单位）→ 反向补偿 **1** 档
    - 急升 → +1 档
    - 急降 → -1 档
  - 此逻辑始终触发，目的是在温度快速变化时提前干预，抑制过冲/过冷

- **防重复**：
  - 记录上次调整时的电池温度（`prev_batt_temp`）
  - 如果当前温度与该值相同，跳过本次升降档
  - 死区内也更新该记录，避免反复进入计算

- **冷却期已移除**：已删除所有冷却期逻辑（包括 CPU 紧急的冷却）。

## CPU 紧急干预细节

- 使用一阶低通滤波：`weighted = raw × CPU_FILTER_ALPHA% + old × (100-CPU_FILTER_ALPHA)%`
- 首次运行直接赋值，不做滤波
- 滤波系数默认 25 (α=0.25)，可通过 `CPU_FILTER_ALPHA` 配置

**紧急等级**（阈值均可通过 profile.conf 调整）：

| 等级 | 触发条件（默认） | 强制最低档位（默认） |
|------|----------------|-------------------|
| 0 | 正常 | 无 |
| 1 | CPU > 65°C (`CPU_EMERG_1`) | 5 (`EMERG_FORCED_1`) |
| 2 | CPU > 75°C (`CPU_EMERG_2`) | 7 (`EMERG_FORCED_2`) |
| 3 | CPU > 85°C (`CPU_EMERG_3`) | 9 (`EMERG_FORCED_3`) |

**退出紧急时**限制电池档位上限（在同步之后执行，保证生效）：
- 仅在电池温度低于升档阈值（`BATT_BASELINE + BATT_ZONE_1`=35.7°C）时执行
- 如果电池温度仍高于升档阈值，说明散热需求仍在，不降档（防止 CPU 温度反弹）
- 满足条件时：
  - 退出等级 3 → `battery_fan_level ≤ EMERG_FORCED_3`
  - 退出等级 2 → `battery_fan_level ≤ EMERG_FORCED_2`
  - 退出等级 1 → `battery_fan_level ≤ EMERG_FORCED_1`

## 启动流程

1. 设定默认日志路径（根据二进制文件名）
2. 加载配置（`--config` 参数 > 自动检测 > 全部默认值）
3. 读电池温度计算初始档位
4. 进入工作模式后每 5 秒执行 `main_loop()`
   - `main_loop` 中通过 `pgrep` 检测 App 进程存活
   - App 未运行时等待复活，复活后强制重发当前档位

## 注意事项

- **温度传感器路径** `thermal_zone30~40` 是 K60 机型值，其他 SoC 可能需要调整扫描范围
- **去重逻辑**：`apply_level()` 缓存上次参数，连续相同不下发，但 App 复活时强制清除缓存重发
- **`patch_tls.py`**：NDK 静态编译后需要执行，修复 PT_TLS 对齐；无 TLS 段时返回非零，需要 `|| echo "skipped"` 容忍
- **`#include <stdarg.h>`**：`write_log()` 用了 `va_list`，NDK clang 下不是隐式包含的
- **FIFO 已废弃**：原 FIFO 通信（`/data/local/tmp/b6x_fifo`）已删除，改用 `pgrep` 检测 App 进程
- **KernelSU noexec**：KernelSU 模块目录挂载了 `noexec`，`service.sh` 会将二进制复制到 `/data/local/tmp/` 执行
