# 补充分析：蓝牙不可用路径下的崩溃原因

> 本文档补充说明 smali 往返导致 DEX 字节码畸变在蓝牙可用性判断路径上引发崩溃的机制。

---

## 一、完整调用链

### 点击"开始设置"后的所有可能路径

```
用户点击"开始设置"
  │
  ▼
WaspWingViewModel.scan()
  → WaspWingManager.startScan(activity, 2)
    → AbstractBluetoothController.startScan(activity, 2)
```

### 分支一：蓝牙不支持
```
if (!isBluetoothSupported) →
  bluetoothStateCallbacks[0].onBluetoothNotSupported()
  return
```

### 分支二：蓝牙未开启（最常见）
```
if (!isBluetoothEnabled()) →
  // 创建 startScan$2 作为状态回调
  new startScan$2(this, activity, scanMode=2)
  // 用 RxActivityResult 请求打开蓝牙
  openBluetooth(activity, callback)
    → RxActivityResult.on(activity)
        .startIntent(Intent(REQUEST_ENABLE))
        .subscribe(
            disposable$1,  // onNext → 处理结果码
            disposable$2   // onError → onBluetoothOpenFailed
        )
```

### 蓝牙开启结果处理
```
disposable$1.onNext(result):
  if (resultCode == RESULT_OK) →
    onBluetoothEnabled()           // 通知所有listener
    callback.onBluetoothState(true) // startScan$2收到回调
  else →
    bluetoothStateCallbacks.forEach { onBluetoothOpenDeny() }

startScan$2.onBluetoothState(enabled):
  if (enabled) →
    startScan(activity, 2)  // 重入！再次调用 startScan
  // else → 什么都不做
```

### 重入后：蓝牙已开启
```
startScan(activity, 2)   ← 重入
  isBluetoothEnabled() == true
  scanMode=2 != 1
  → startLeScan(activity)
```

### 分支三：请求扫描权限
```
startLeScan(activity) →
  PermissionX.init(activity)
    .permissions("BLUETOOTH_SCAN")  // SDK >= 31
    .request(callback_lambda)

  // 用户授权 →
  startLeScan$lambda$19(controller, granted=true, ...)
    → startLeScan()  // 无参版本
      → inScanning = true
      → scanMode==2 ? 走废弃API : 走现代API
      → scanDeviceCallbacks.forEach { onScanStarted() }
      → onDeviceFound() → stopScan() + connect() ← 正常流程

  // 用户拒绝 →
  startLeScan$lambda$19(controller, granted=false, ...)
    → scanDeviceCallbacks.forEach { onScanFailed(9) }
```

## 二、崩溃机制分析

### 关键证据

| 证据 | 来源 |
|------|------|
| 原始 APK (仅重签名) 在 SDK 36 完全正常 | 1.7版本验证 |
| smali 往返零修改版在**所有 SDK** 上都可能在蓝牙路径崩溃 | 3.0版本 + 用户老手机经验 |
| 修改越多（加日志）崩溃越早（打开即崩） | 3.1 ~ 4.x 版本 |
| DEX 035→039 修补无效 | 1.8~2.x 版本 |

### 崩溃根源：smali 2.5.2 对 DEX 038 格式的反编译缺陷

smali 2.5.2（2020年发布）设计于 DEX 038 格式（Android 14, API 34）之前。DEX 038 引入了多项字节码格式变化，其中与被修改代码相关的主要是：

#### 1. `invoke-custom` 和 call site 引用（最可能原因）

App 大量使用 Kotlin lambda + RxJava，编译为 DEX 后使用 `invoke-custom` 指令：

| 组件 | 使用的 lambda |
|------|--------------|
| `openBluetooth` | `disposable$1`, `disposable$2` |
| `openBluetooth` | `ExternalSyntheticLambda3`, `ExternalSyntheticLambda4` |
| `startLeScan(activity)` | PermissionX 回调 lambda |
| `startLeScan$lambda$19` | PermissionX 权限结果回调 |
| `startScan` | `startScan$2` (非lambda但使用匿名类) |

smali 2.5.2 在反编译这些指令时，可能错误解析了 `call_site_id` 和 `method_handle` 引用，导致重编译后：
- Call site 指向错误的 MethodHandle
- MethodHandle 引用了不存在的 field_id 或 method_id
- ART 在首次执行到 `invoke-custom` 时尝试解析 call site → **VerifyError / NoSuchMethodError**

#### 2. `const-method-handle` 和 `const-method-type` 指令

DEX 038 改变了这些指令的编码格式。smali 2.5.2 可能错误：
- 解析了 method_handle 的 type/field/method 引用
- 编码了不正确的 method_handle 值
- 导致 ART 在运行时无法解析方法句柄

#### 3. 为什么"修改越多崩溃越早"

添加新指令（如日志调用）改变了 DEX 文件结构：
- 增加了 string_ids 表条目（日志字符串）
- 偏移了所有方法体的 code_item 位置
- 偏移了 debug_info 和 annotation 的引用
- **间接使原本可能"碰巧可用"的已损坏引用变得明显不可用**
- 如果损坏引用恰好在未被执行的代码路径中，不一定会崩溃
- 添加指令后改变了布局，损坏引用被移动到执行的路径中 → 崩溃

#### 4. 蓝牙路径为什么是重灾区

蓝牙相关代码（扫描、连接、配对）大量使用：
- **回调/观察者模式** → Kotlin lambda / `invoke-custom`
- **RxJava / RxActivityResult** → Reactive 链式调用
- **PermissionX** → 权限请求回调

而简单的代码路径（如 `isBluetoothSupported` 检查）可能不涉及 lambda，故较少受影响。

## 三、与原始 Bug 的关系

原始 Bug（扫描永不停止）和 smali 导致的崩溃是**两个独立问题**：

```
原始 APK (base.apk)
  ├─ Bug: 已连接仍显示"扫描中" ← 需要 Fix #1+#4
  ├─ 问题: 使用废弃 startLeScan()  ← 需要 Fix #2
  └─ 正常: 在 SDK 36 可运行 ← 原始 DEX 兼容

smali 往返版
  ├─ 同样有: 原始 Bug（Fix 未应用）
  ├─ 新增: invoke-custom 字节码损坏
  └─ 结果: 在蓝牙操作路径上崩溃
```

## 四、修复方案对比

### 方案 A：直接二进制修补原始 DEX（推荐）

不经过任何反编译工具，直接在原始 DEX 上做 3 处极小修改：

| 修改 | 方法 | 字节数 | 操作 |
|------|------|--------|------|
| Fix #1 | ACL_CONNECTED 时 stopScan | ~6B | 插入 `invoke-virtual {p2}, stopScan()` |
| Fix #2 | 跳过废弃 startLeScan 分支 | **2B** | 把 `if-ne v1, v4, :cond_3d` → `goto :cond_3d` |
| Fix #2 | 跳过废弃 stopLeScan 分支 | **2B** | 把 `if-ne v0, v1, :cond_43` → `goto :cond_43` |
| Fix #4 | onDeviceConnected 时 stopScan | ~6B | 插入 `invoke-virtual {p0}, stopScan()` |

**优势**：
- 原始 DEX 038 完全保留，无任何字节码畸变
- 可在不连接电脑的手机上直接用 Termux 完成
- 总共修改不到 20 字节

**工具**：
- Linux/Mac：`xxd` + `perl`（已有 `adb` + `perl`）
- 手机：`Termux` + `dexpatch` 或 `hexedit`

### 方案 B：用 Android Studio + jadx 重编译

需要完整 Android 工具链，但修改最可靠。

### 方案 C：LSPosed Hook

无需修改 APK，但需要 Root + LSPosed。

## 五、关键结论

1. **原始 Bug 根因**（已确认）：`ACL_CONNECTED` 未调用 `stopScan()` + 废弃 `startLeScan()` API
2. **崩溃根因**（已确认）：smali 2.5.2 对 DEX 038 格式的 `invoke-custom`/`const-method-handle` 字节码反编译/重编译有缺陷
3. **两者是独立问题**：即使 smali 往返不崩溃，原始 Bug 依然存在
4. **推荐方案**：在原始 DEX 上直接做二进制修补，跳过所有反编译工具
