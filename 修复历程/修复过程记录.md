# 飞智散热器 App 修复过程记录

> 日期：2026-06-03 ~ 2026-06-04
> 目标：修复 base.apk 在 Android 16 (SDK 36) 上的"已连接蓝牙但持续显示扫描中"问题

---

## 一、原始 Bug 分析

### Bug 表现
App 显示蓝牙已连接，但 UI 持续显示"扫描中"，无法进入控制界面。

### 反编译
- 使用 baksmali 2.5.2 将 classes.dex / classes2.dex 反编译为 smali 代码
- 包名：`com.flydigi.waspwing.experimental`
- App 名：飞智散热器 (WaspWing)

### 代码结构
```
com.example.extool              ← 应用层
  BluetoothViewModel.smali       ← 抽象 ViewModel
  WaspWingViewModel.smali        ← 散热器具体实现

com.flydigi.sdk.bluetooth        ← 蓝牙 SDK
  AbstractBluetoothController    ← 核心控制器（扫描/连接/配对）
  LeDataInteractionController    ← BLE 数据交互
  BtDataInteractionController    ← 经典蓝牙数据交互

com.flydigi.sdk.waspwing         ← 散热器协议层
  WaspWingManager                ← 管理器单例
  WaspWingDataInteractionController ← BLE 交互实现
```

### 根因定位
详见 `反编译及分析结果/bug_analysis_report.md`

**核心问题**：当已配对设备通过系统自动回连时，`ACL_CONNECTED` 广播触发了 `onDeviceConnected()`，但**从未调用 `stopScan()`**。

`stopScan()` 只在两处被调用：
- `BluetoothViewModel.onDeviceFound()` → 扫描发现新设备
- `AbstractBluetoothController.createBond()` → 用户手动配对

已配对设备自动回连时，两条路径都不触发 → 扫描永不停止。

### SDK 36 加剧因素
1. **废弃 API**：App 使用 `BluetoothAdapter.startLeScan()` (API 21 废弃)，scanMode=2 硬编码使用废弃路径
2. **registerReceiver 无标志**：4处 `registerReceiver()` 未指定 `RECEIVER_NOT_EXPORTED`
3. **connectDevice()**：SDK>=28 直接 return true 不做实际操作

---

## 二、修复方案设计

| Fix # | 修改内容 | 文件 |
|--------|---------|------|
| 1 | ACL_CONNECTED 时调用 stopScan() | mConnectStateReceiver$1.smali |
| 2 | 废弃 startLeScan/stopLeScan → BluetoothLeScanner | AbstractBluetoothController.smali |
| 3 | registerReceiver 添加 RECEIVER_NOT_EXPORTED | AbstractBluetoothController.smali |
| 4 | onDeviceConnected 时调用 stopScan() | BluetoothViewModel.smali |

## 三、修复尝试时间线

### 阶段 1：直接 smali 修改（1.1 ~ 1.2）
- **1.1**：应用全部 4 个 Fix → 点击"开始设置"闪退
- **1.2**：回退 Fix #3 → 仍然闪退
- 分析：Fix #3 改变 registerReceiver 方法签名，可能导致 DEX 验证失败

### 阶段 2：添加日志诊断（1.3 ~ 1.9）
- **1.3**：新增 logFile 方法写入 `/data/data/com.flydigi.waspwing.experimental/files/` → **打开即闪退**
- **1.4**：更正包名 + Log.e 内联 → 闪退
- **1.5**：仅 Log.e 内联，无新方法 → 闪退

### 阶段 3：发现 DEX 版本问题（1.6 ~ 2.x）
- **关键发现**：原始 APK DEX 版本为 **038**（Android 14），smali 2.5.2 输出为 **035**（Android 13）
- **1.6**（零修改 smali 往返，DEX 035）：打开即闪退
- **1.7**（原始 DEX + 仅重签名）：**正常工作！** ← 证明原始 DEX 在 SDK 36 兼容
- **1.8**（smali 往返 + DEX 039 修补）：打开即闪退
- **2.30~2.36**（不同 `--api` 级别编译）：全部打开即闪退

**结论**：不仅是 DEX 版本号问题，smali 2.5.2 输出的字节码本身在 SDK 36 上有兼容性问题。

### 阶段 4：发现 debug_info 问题（3.0）
- **关键发现**：smali 默认带 debug info，往返后 debug_info 条目从 32,257 暴增到 48,612
- **3.0**（`--di=false` 零修改）：**App 可以打开！** 点击"开始设置"后闪退
- 验证：3.0 用 `--di=false` 成功，之前的版本全是 debug info 过多导致打开即崩

### 阶段 5：日志追踪（3.1 ~ 4.x）
- 尝试在 3.0 基础上加 Log.e → 打开即闪退
- 尝试加 Logger.logd（已有方法引用）→ 多条日志打开即崩，单条日志（4.0）可打开
- **关键发现**：smali 中添加任何新指令都会导致 DEX 稳定性下降

### 阶段 6：修复尝试（5.0 ~ 8.0）
- **5.0**（Fix #1+#2+#4，无日志）：点击"开始设置"后闪退（与 3.0 相同）
- **7.0**（仅 Fix #2，替换废弃 API）：同上
- **8.0**（Fix #2 + 默认 API level + DEX 039 补丁）：同上

**结论**：Fix #2 未解决问题。崩溃不在废弃 API 调用本身，而在 smali 往返引入的字节码畸变。

---

## 四、关键发现总结

| # | 发现 | 影响 |
|---|------|------|
| 1 | smali 2.5.2 输出 DEX 035，SDK 36 需要 038+ | 需修补 DEX 头 |
| 2 | `--debug-info=false` 必须加，否则 debug_info 暴增 | 不加直接打开就崩 |
| 3 | **smali 往返引入字节码畸变** | 零修改也会在扫描流程中崩溃 |
| 4 | 添加新指令（包括日志）会加剧畸变 | 修改越多崩溃越早 |
| 5 | 原始 DEX 直接重签名可正常工作 | 问题 100% 在 smali 工具链 |

## 八、关键命令备忘

```bash
# 反编译（无 debug info）
java -jar baksmali.jar d --di=false -o output_dir classes.dex

# 编译（指定 API level）
java -jar smali.jar a --api 36 -o output.dex input_dir

# DEX 035 → 039 修补（Perl）
perl -e '
use Digest::SHA qw(sha1);
use Compress::Zlib qw(adler32);
# ... (见修复过程1.6中的脚本)
'

# 打包 APK
cd repack_dir && jar cf ../unsigned.apk *

# 签名
jarsigner -sigalg SHA256withRSA -digestalg SHA-256 \
  -keystore debug.keystore -storepass android unsigned.apk debug
```
